## Styling Guidelines
* All examples use Tailwind CSS utility classes.
* No Contember-specific theme classes are required.

## Core Concepts
* `ChangeMyPasswordFormFields` renders all inputs needed to change the current user’s password.
* It must be nested inside `ChangeMyPasswordForm`, which supplies mutation logic and validation.
* Wrap the fields in a native `<form>` element to enable browser submission UX.
* You can style the surrounding layout freely with Tailwind classes.

## Quick Start
```tsx
import { ChangeMyPasswordForm, ChangeMyPasswordFormFields } from '~/lib/tenant'

export default function PasswordPage() {
	return (
		<ChangeMyPasswordForm>
			<form className="grid gap-4 max-w-md">
				<ChangeMyPasswordFormFields />
			</form>
		</ChangeMyPasswordForm>
	)
}
```

## Examples

### 1. Toast notification on success
```tsx
import { ChangeMyPasswordForm, ChangeMyPasswordFormFields } from '~/lib/tenant'
import { useShowToast, ToastContent } from '~/lib/ui'

export function SecuritySection() {
	const showToast = useShowToast()

	return (
		<ChangeMyPasswordForm onSuccess={() =>
			showToast(<ToastContent>Password changed</ToastContent>, { type: 'success' })
		}>
			<form className="grid gap-4">
				<ChangeMyPasswordFormFields />
			</form>
		</ChangeMyPasswordForm>
	)
}
```
This pattern shows how to surface API feedback to the user.

Combine `ChangeMyPasswordFormFields` with your design system’s `Card` component to keep settings pages consistent.

## Best Practices & Gotchas
* Always place the fields inside `ChangeMyPasswordForm`; otherwise, mutations will fail.
* Use semantic `<form>` and `<button type="submit">` elements to maintain accessibility.
* Apply Tailwind spacing (`gap-*`) on the `<form>` wrapper instead of individual fields.
* Do not pass unknown props; the component ignores them.
* Handle success and error states via `onSuccess` and `onError` props on `ChangeMyPasswordForm`.
