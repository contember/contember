## Overview
DefaultRepeater wraps `Repeater` and renders a ready‑made UI for managing a list of entities.  
It handles ordering, add/remove buttons, and item actions so you can focus on form fields.

## Styling Guidelines
* All snippets use TailwindCSS utility classes.  
* No Contember‑specific theme classes are required; integrate with your existing Tailwind setup.

## Core Concepts
* DefaultRepeater manages an entity collection for you.  
* You can make the list sortable by passing `sortableBy`.  
* Place form fields inside DefaultRepeater to edit each item.  
* Use `RepeaterItemActions` to place per‑item controls (e.g., delete).  
* The component forwards every prop of the underlying `Repeater`.

## Quick Start
```tsx
import { DefaultRepeater, RepeaterItemActions, RepeaterRemoveItemButton } from '~/lib/repeater'

export const NotesForm = () => (
	<DefaultRepeater entities="Note" sortableBy="order">
		<InputField field="title" className="w-full" />
		<RepeaterItemActions>
			<RepeaterRemoveItemButton className="text-red-600" />
		</RepeaterItemActions>
	</DefaultRepeater>
)
```

## Props Reference
| Name | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| entities | string | No | – | Entity name for a standalone collection. |
| field | string | No | – | Relation field when nested in `EntitySubTree`. |
| orderBy | string | No | – | Static ordering for non‑sortable lists. |
| sortableBy | string | No | – | Column used when drag‑and‑drop sorting is enabled. |
| title | string | No | – | Heading displayed above the list. |
| addButtonPosition | string | No | `around` | Where the “Add” button appears (`start`, `end`, `around`). |
| initialEntityCount | number | No | 0 | Entities created on first render. |
| ...RepeaterProps (inherited) | RepeaterProps | No | – | All props from the base `Repeater` component. |

## Sub-Components / Building Blocks
### RepeaterItemActions
Wrap per‑item buttons (e.g., delete or custom menus).

### RepeaterRemoveItemButton
Renders a delete button inside `RepeaterItemActions`.

## Examples
### Sortable list with relation field
```tsx
<DefaultRepeater field="items" sortableBy="order" title="Products">
	<InputField field="name" className="w-full" />
	<SelectField field="category">
		<Field field="name" />
	</SelectField>
	<RepeaterItemActions>
		<RepeaterRemoveItemButton />
	</RepeaterItemActions>
</DefaultRepeater>
```
The list is sortable by the `order` column. Each item edits its `name` and related `category`.

### Pre‑filled list with initial entity
```tsx
<DefaultRepeater field="notes" initialEntityCount={1}>
	<TextAreaField field="content" className="w-full" />
</DefaultRepeater>
```
A single blank entity is inserted automatically, saving one click for users.

### Non‑sortable static order
```tsx
<DefaultRepeater entities="TodoItem" orderBy="createdAt DESC">
	<InputField field="task" />
	<RepeaterItemActions>
		<RepeaterRemoveItemButton />
	</RepeaterItemActions>
</DefaultRepeater>
```
Items follow the `createdAt` order delegated to the backend; users cannot reorder them.

## Best Practices & Gotchas
* Always choose either `sortableBy` or `orderBy`, not both.  
* Nest DefaultRepeater under `EntitySubTree` when editing relations.  
* Wrap destructive buttons in `RepeaterItemActions` for consistent spacing.  
* Remember to persist the form with `PersistButton` to save ordering changes.  
* Large lists may benefit from virtual scrolling; DefaultRepeater does not handle this.
