# RepeaterItemActions

## Overview
`RepeaterItemActions` wraps buttons or menus that act on a single repeater item.
Place it in the upper-right corner of each item to keep actions consistent and discoverable.

## Styling Guidelines
* All snippets use Tailwind CSS.
* The default Contember admin theme already styles the container; no extra theme classes are required.

## Core Concepts
* Repeater items need a unified action area.
* `RepeaterItemActions` renders its children inline.
* Combine it with `RepeaterRemoveItemButton` or custom dropdowns.
* Position the component inside a `DefaultRepeater` item layout.

## Quick Start
```tsx
import { DefaultRepeater, RepeaterItemActions, RepeaterRemoveItemButton } from '~/lib/repeater'

<DefaultRepeater entities="Task" title="Tasks">
  {/* Your fields here */}
  <RepeaterItemActions className="flex gap-2">
    <RepeaterRemoveItemButton />
  </RepeaterItemActions>
</DefaultRepeater>
```

## Sub-Components / Building Blocks

### RepeaterRemoveItemButton
Built-in button that deletes the current repeater item. Place it inside `RepeaterItemActions`.

## Examples

### 1 — Sortable repeater on a collection
```tsx
<DefaultRepeater entities="RepeaterItem" sortableBy="order" title="Foo items" addButtonPosition="around">
  <InputField field="title" />

  <SelectField field="relation">
    <Field field="id" /> /
    <Field field="name" />
  </SelectField>

  <RepeaterItemActions className="flex gap-2">
    {repeaterDropdown}
    <RepeaterRemoveItemButton />
  </RepeaterItemActions>
</DefaultRepeater>
```
`sortableBy="order"` adds drag handles. The action area stays fixed, so users always know where to click.

### 2 — Repeater on a relation subtree
```tsx
<EntitySubTree entity="RepeaterRoot(unique = unique)" setOnCreate="(unique = unique)">
  <DefaultRepeater field="items" sortableBy="order" title="Foo items" addButtonPosition="around">
    <Field field="id" /><br/>
    <InputField field="title" />
    <SelectField field="relation">
      <Field field="id" /> /
      <Field field="name" />
    </SelectField>

    <RepeaterItemActions className="flex gap-2">
      {repeaterDropdown}
      <RepeaterRemoveItemButton />
    </RepeaterItemActions>
  </DefaultRepeater>
</EntitySubTree>
```
`RepeaterItemActions` works the same inside an `EntitySubTree`; no extra wiring is necessary.

### 3 — Non-sortable repeater
```tsx
<DefaultRepeater entities="RepeaterItem" orderBy="order" title="Foo items">
  <Field field="title" />

  <RepeaterItemActions className="flex gap-2">
    {repeaterDropdown}
    <RepeaterRemoveItemButton />
  </RepeaterItemActions>
</DefaultRepeater>
```
When the list is static (`orderBy` only), the action area remains the interaction hub for each row.

### 4 — Minimal removal-only actions
```tsx
<RepeaterItemActions>
  <RepeaterRemoveItemButton />
</RepeaterItemActions>
```
A single destructive action is common for simple lists. The default spacing is sufficient, so `className` is optional.

### 5 — Pipeline management repeater
```tsx
<Binding>
  <Slots.Title>Pipeline</Slots.Title>
  <Slots.Actions>
    <PersistButton />
  </Slots.Actions>

  <DefaultRepeater entities="Pipeline" orderBy="branch.name">
    <PipelineForm />

    <RepeaterItemActions className="flex gap-2">
      <RepeaterRemoveItemButton />
    </RepeaterItemActions>
  </DefaultRepeater>
</Binding>
```
This production example shows `RepeaterItemActions` inside a complex form. The action area stays lightweight, keeping the form focused.

## Best Practices & Gotchas
* Keep only essential controls in the action area to avoid clutter.
* Use `className="flex gap-2"` for predictable spacing.
* Avoid nesting complex forms inside `RepeaterItemActions`; keep interactions simple.
* Pair destructive actions with confirmation if data loss is possible.
