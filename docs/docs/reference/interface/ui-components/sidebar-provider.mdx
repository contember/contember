# SidebarProvider

## Overview
SidebarProvider wraps your layout and shares sidebar state through React context.  
Place it high in the tree to let child components open, close, or read the sidebar status.

## Styling Guidelines
* All snippets use Tailwind utility classes.  
* The provider itself adds no DOM, so you only style the visible sidebar components.

## Core Concepts
* Context Provider – exposes sidebar visibility to descendants.  
* Composition – you decide which Sidebar, triggers, or slots to render.  
* Mobile support – combine with SidebarMobileTrigger for responsive layouts.

## Quick Start
```tsx
import { SidebarProvider } from '~/lib/ui/sidebar'

export const AppLayout = () => (
	<SidebarProvider>
		{/* your sidebar */}
		{/* your content */}
	</SidebarProvider>
)
```

### Responsive Layout With Toggle
```tsx
import { SidebarProvider } from '~/lib/ui/sidebar'
import { SidebarMobileTrigger } from '~/lib/ui/sidebar'
import { MenuIcon } from 'lucide-react'

export const ResponsiveLayout = ({ children }: { children: React.ReactNode }) => {
	const [open, setOpen] = useState(true)

	return (
		<SidebarProvider>
			{open && <AppSidebar className="hidden lg:block" />}
			<header className="flex items-center gap-2 p-4 border-b">
				<SidebarMobileTrigger
					icon={<MenuIcon />}
					onClick={() => setOpen(!open)}
					className="lg:hidden"
				/>
				<h1 className="text-lg font-semibold">Dashboard</h1>
			</header>

			<main className="p-4">{children}</main>
		</SidebarProvider>
	)
}
```
The provider holds the context; the trigger toggles `open`. On large screens, the sidebar stays visible. On small screens, users touch the burger icon.

## Best Practices & Gotchas
* Wrap the whole page, not just the sidebar, so any descendant can toggle visibility.  
* Keep SidebarProvider singular—nesting providers may cause inconsistent state.  
* Manage mobile and desktop visibility in one place to avoid duplication.
