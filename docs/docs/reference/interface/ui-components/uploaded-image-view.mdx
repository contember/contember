## Styling Guidelines
* All examples use Tailwind CSS utility classes.  
* The component adds no built-in styles, so you may combine it with your Contember theme classes if needed.

## Core Concepts
* Map CMS file fields to visual output with `urlField` and optional metadata fields.  
* Wrap the component in `<HasOne>` (or another binding component) so field values come from the current entity.  
* Optional action elements (`actions`, `edit`, `DestroyAction`) let you build inline editing UIs.  
* The component only renders; it never mutates data unless you supply custom actions.

## Quick Start
```tsx
import { HasOne } from '@contember/admin'
import { UploadedImageView } from '~/lib/upload'

<HasOne field="image">
  <UploadedImageView urlField="url" className="max-w-full rounded shadow" />
</HasOne>
```

## Best Practices & Gotchas
* Supply only fields you actually store; missing props do not break rendering.  
* Always wrap the component in a binding context (`HasOne`, `FieldView`) so data loads.  
* Use `noDestroy` when the file should survive record deletion in the UI.  
* Tailwind’s `object-cover` class yields consistent thumbnails.  
* Pass a custom `DestroyAction` if you need confirmation dialogs or a disconnect action.  
* Overlay elements (checkboxes, buttons) should share a `relative` parent to avoid stacking issues.  
* Hide default danger controls with scoped selectors when you provide your own UI.
